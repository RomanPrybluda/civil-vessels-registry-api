# Project Errors To Avoid

Цель файла: фиксировать типовые ошибки, которые уже встречались в проекте, чтобы не повторять их.

## Исторические ошибки и как их избегать

### 1) Несогласованная структура маршрутов
- Проблема: часть эндпоинтов жила без общего префикса или с разными алиасами.
- Последствие: путаница в клиенте, Swagger и README.
- Правило:
  - Все маршруты только под `/api`.
  - Для классификационных обществ использовать `class-societies` во всех слоях.

### 2) Слабая валидация входных DTO
- Проблема: недостаточно строгие ограничения в create/update DTO.
- Последствие: некорректные значения доходят до бизнес-логики и БД.
- Правило:
  - Для каждого поля задавать явные `class-validator` ограничения (тип, длина, диапазон, формат URL и т.д.).
  - Не полагаться только на DB constraints.

### 3) Неявная/избыточная авторизация
- Проблема: риск размытых границ между `user`, `manager`, `admin`.
- Последствие: возможный лишний доступ к write-операциям.
- Правило:
  - `GET` остаются публичными.
  - `manager` пишет только `vessels` (`POST/PUT`).
  - Полный write-доступ только у `admin`.
  - Держать в актуальном состоянии `authorization-matrix.spec.ts`.

### 4) Mock-данные аутентификации в runtime
- Проблема: fallback-пользователи или тестовые креды в боевом коде.
- Последствие: снижение безопасности и непредсказуемая конфигурация.
- Правило:
  - `AUTH_USERS_JSON` обязателен.
  - В production-ready коде не хранить mock credentials.

### 5) Healthcheck без проверки зависимости БД
- Проблема: endpoint мог отвечать `200`, даже когда БД недоступна.
- Последствие: ложноположительное состояние сервиса.
- Правило:
  - `GET /api/health` всегда включает DB ping и возвращает `503`, если БД недоступна.

### 6) Рассинхрон между кодом и документацией
- Проблема: изменения в API/ролях/маршрутах вносились без синхронизации docs.
- Последствие: документация перестает отражать реальное поведение.
- Правило:
  - После каждой задачи проверять: `README.md`, `doc/CODEX_MEMORY.md`, файлы в `docs/`.
